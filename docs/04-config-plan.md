# 04 配置管理文档

## 1. 配置项范围
- README
- 项目启动说明
- WBS 文档
- 进度计划文档
- 总结报告
- 示例代码或资源文件

## 2. 分支策略
- main
- dev
- feature-* 

## 3. 命名约定
请填写文档命名、分支命名、提交说明等约定。
#### 文档命名：
- 统一前缀：01-、02-、03-、05-
- 统一格式：小写字母、中横线分隔，.md 后缀
- 示例：01-wbs.md、02-schedule.md
#### 分支命名：
- main：主分支
- dev：开发分支
- feature-功能名称：功能分支
- 示例：feature-docs、feature-conflict
#### 提交说明规范：
- feat: 新增文档或功能
- docs: 更新 / 完善文档
- fix: 修复问题、解决冲突
- merge: 分支合并
- 示例：docs: 完成三级WBS分解
## 4. 合并规则
请说明谁负责合并，何时合并，出现冲突时如何处理。
#### 1.合并责任人：韩宝磊（配置管理员、项目经理）
#### 2.合并时机
- 功能分支完成后 → 合并到 dev
- 全部成果完成、冲突解决后 → dev 合并到 main
#### 3.冲突处理规则
- 禁止自动覆盖，必须手动解决冲突
- 保留有效内容，删除 Git 冲突标记
- 解决后重新提交，保留完整操作记录
## 5. 版本留痕
请写明本组如何保留阶段性版本与关键提交记录。
#### 1.所有阶段性成果必须通过 git commit 提交，不允许直接上传文件。
#### 2.关键节点（初始化、WBS 完成、进度计划、冲突解决、总结报告）必须单独提交。
#### 3.每次提交信息清晰可追溯，提交人统一为 hanbaolei。
#### 4.使用 git log 保留完整提交历史，用于实验核查与评分。
#### 5.最终版本必须合并并推送到 main 分支，形成最终可验收版本。
